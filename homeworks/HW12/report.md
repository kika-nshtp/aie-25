# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- **Датасет:** `S12-hw-dataset.csv` — почасовой временной ряд `target` за период 2025-01-01 – 2025-06-29 (4320 наблюдений). Прогнозировалось следующее значение ряда (горизонт h=1).
- **Temporal split:** данные разбиты строго по хронологии — 70% на обучение, 15% на валидацию, 15% на тест.
- **Подходы:** B1 (naive-last), B2 (moving-average за 24 часа), B3 (Ridge на лаговых/rolling/календарных признаках), R1 (GRU на скользящих окнах длиной 24).
- **Лучший подход по validation:** B3 (Ridge, val MAE = 4.41) — линейная модель на признаках из прошлого уверенно обогнала все остальные варианты на validation. GRU показал результат чуть хуже наивного baseline, что характерно для коротких рядов с сильной локальной инерцией.

## 3. Данные и постановка задачи

- **Датасет:** `S12-hw-dataset.csv`
- **Целевая колонка:** `target` (числовые значения, диапазон ~69–210)
- **Частота временного ряда:** почасовая (1 наблюдение в час)
- **Горизонт прогноза:** 1 шаг вперёд (следующий час)
- **Размер датасета:** 4320 строк
- **Диапазон дат:** 2025-01-01 00:00 → 2025-06-29 23:00
- **Дополнительные внешние признаки:** нет (только `date` и `target`)
- **Комментарий:** ряд имеет выраженную суточную сезонность — значения регулярно поднимаются и опускаются в течение дня. Присутствуют отдельные выбросы. Пропусков нет. Долгосрочный тренд слабый, что делает задачу умеренной сложности — лаговые признаки хорошо захватывают локальную инерцию ряда.

---

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- **Схема:** `train` → `validation` → `test` строго в хронологическом порядке, без перемешивания строк.
- **Доли:** 70% / 15% / 15%.
- **Границы:**
  - train: 2025-01-01 → 2025-04-21 (3024 строки)
  - validation: 2025-04-21 → 2025-05-28 (648 строк)
  - test: 2025-05-28 → 2025-06-29 (648 строк)
- **Почему `random split` некорректен:** при случайном перемешивании модель обучается на «будущих» значениях и тестируется на «прошлых». Лаговые признаки тогда содержат информацию, которая в реальности ещё недоступна. Это приводит к оптимистичным метрикам, не отражающим реальную прогнозную способность.

### 4.2. Признаки для baseline-моделей

- **Лаги:** `lag_1`, `lag_7`, `lag_14`, `lag_24`, `lag_168` (1 ч, 7 ч, 14 ч, 1 сутки, 1 неделя назад).
- **Rolling-признаки:** `rolling_mean_7`, `rolling_std_7`, `rolling_mean_24`, `rolling_std_24` — всегда с `shift(1)` перед rolling, чтобы исключить текущее значение из окна.
- **Календарные признаки:** `hour`, `dayofweek`, `month`, а также синус/косинус-кодирование часа (`hour_sin`, `hour_cos`) и дня недели (`dow_sin`, `dow_cos`).
- **Обработка пропусков:** первые строки датафрейма неизбежно теряются при построении лагов (lag_168 = 168 строк). Они удаляются через `dropna()` — это корректно, так как без полного контекста лаговые признаки ненадёжны.
- **Масштабирование:** `StandardScaler` обучается только на `X_train`. `X_val` и `X_test` масштабируются через `.transform()` — без доступа к их статистикам.

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (naive-last):** прогноз = значение `lag_1` (последний известный час). Никакого обучения.
- **B2 (moving-average):** прогноз = `rolling_mean_24` (среднее за последние 24 часа, вычислено с `shift(1)`). Никакого обучения.
- **B3 (ridge-lag-features):** `Ridge(alpha=1.0)` на 16 признаках — лаги, rolling mean/std, календарные. Scaler: `StandardScaler` (fit только на train).
- **R1 (gru-forecast):** `GRU(hidden=64, layers=2)` на скользящих окнах длиной 24. Вход: нормализованный ряд `target`. Выход: следующее значение.

**Дополнительно:**
- Основная метрика выбора лучшей модели: **val MAE**
- Дополнительные метрики: RMSE, MAPE
- `window_size` для GRU: **24** (24 часа = одни сутки)
- Batch size: **64**
- Максимальное число эпох: **30**
- Optimizer: **Adam**
- Learning rate: **5e-4**
- Как сохранялась лучшая модель: `state_dict()` при минимальном `val_loss` → `artifacts/best_gru.pt`

## 7. Анализ

Наивный baseline (B1) оказался неожиданно сильным конкурентом: MAE = 6.40 на validation, что лучше, чем у GRU (5.79) и MA-24 (13.35). Это типично для почасовых рядов с сильной локальной инерцией — следующий час часто близок к текущему. Moving average за 24 часа (B2) слишком «размывает» информацию и является худшим подходом: он сглаживает пики и теряет точность именно в моменты изменений.

Ridge на лаговых признаках (B3) показал лучший результат, потому что явно включает `lag_1` (вчерашний час), `lag_24` (тот же час вчера) и `lag_168` (тот же час неделю назад) — три наиболее коррелированных с целевой переменной лага для почасового ряда. Календарные признаки добавляют знание о суточной и недельной сезонности. Линейная комбинация этих признаков уже хорошо описывает ряд.

GRU не смогла превзойти Ridge: ряд относительно короткий (≈3000 точек на обучение), а нейросеть нуждается в большем количестве данных для обобщения. Кроме того, Ridge получает явно инженерные признаки с нужными лагами, тогда как GRU вынуждена извлечь эту информацию из окна самостоятельно за ограниченное число эпох. Кривые обучения GRU показывают, что val loss не уменьшается стабильно — есть колебания, признак недостаточного количества данных.

Суточная сезонность хорошо видна на графике split: значения регулярно повторяют форму. Нестационарность минимальна, что объясняет сравнительно низкую абсолютную ошибку всех моделей (MAPE лучшей модели ≈ 3%).

Возможные источники утечки данных в такой задаче: centered rolling (`center=True`); fit scaler на всём датасете; использование `lag_0` (текущего значения); оценка и выбор модели по test вместо validation. Все эти ошибки были намеренно исключены в данной работе.

Принципиальность выбора по validation заключается в том, что многократные прогоны под разные архитектуры с оценкой на test создают «утечку» информации из теста в выбор модели (multiple comparison problem). Test должен имитировать реальные условия — данные, которые мы увидим только один раз.

---

## 8. Итоговый вывод

Для данного почасового ряда Ridge на лаговых и rolling-признаках является наиболее эффективным и интерпретируемым подходом. Его достаточно в случаях, когда ряд имеет выраженную сезонность и инерцию, а объём данных ограничен (менее 10 тысяч точек). GRU оправдывает себя при значительно большем объёме данных, нелинейных зависимостях и многомерном входе (несколько рядов одновременно). Главный вывод о корректной валидации: temporal split — не опция, а обязательное условие; любая другая схема разбиения для задачи прогноза даёт метрики, несовместимые с реальной прогнозной способностью модели.
